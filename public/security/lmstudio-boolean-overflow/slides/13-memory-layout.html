What lives right after swa_layers

src/llama-model.h
struct llama_model {
  llm_type type;
  llm_arch arch;
  std::string name;

  llama_hparams hparams = {};   // contains swa_layers — the OOB source
  llama_vocab   vocab;          // first victim — right after hparams
  // ...
  std::unordered_set<
    llama_adapter_lora *> loras; // further in — walked by ~llama_model
  // ...
  ~llama_model();
};
src/llama-vocab.h
struct llama_vocab {
  // ...
private:
  struct impl;
  std::unique_ptr<impl> pimpl;  // the ONLY state — 8 bytes
};

Adjacency is destiny vocab sits immediately after hparams, and llama_vocab is a thin wrapper around a single 8-byte pointer.

So the byte offset from swa_layers to vocab.pimpl is a compile-time constant — same in every build, no info-leak required.

Keep going and the OOB writes reach loras — the set the destructor iterates. We will corrupt both with one loop.
