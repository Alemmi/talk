The fix — two minimally invasive changes

1. Hard-fail at the read site (mirror n_expert)
ml.get_key(LLM_KV_BLOCK_COUNT, hparams.n_layer);
if (hparams.n_layer > LLAMA_MAX_LAYERS) {
  throw std::runtime_error(
    "invalid block_count: exceeds LLAMA_MAX_LAYERS");
}
2. Defence in depth — clamp the loop
// set_swa_pattern
const uint32_t n =
  std::min(n_layer, (uint32_t) LLAMA_MAX_LAYERS);
for (uint32_t il = 0; il < n; ++il) { /* ... */ }

Fix #1 Rejects the malicious file the moment block_count is read — the same pattern already guards n_expert.

Fix #2 Contains any future unchecked path into the loop, even if a new caller forgets to validate.

Audit the rest Every other LLAMA_MAX_LAYERS-sized array indexed by n_layer — n_head_arr, n_head_kv_arr, n_ff_arr, recurrent_layer_arr, xielu_*, swiglu_clamp_* — shares the bug.
