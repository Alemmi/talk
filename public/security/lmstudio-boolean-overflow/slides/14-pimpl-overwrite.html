Two ones make a pointer — and only one pointer is possible

every slot is 0 or 1 → only 4 reachable 64-bit values
[hi u32][lo u32]   64-bit value          lands in
  0       0        0x00000000`00000000   the NULL page
  0       1        0x00000000`00000001   the NULL page
  1       0        0x00000001`00000000   ~4 GB region
  1       1        0x00000001`00000001   ~4 GB region  ◄ we use this
so we steer two adjacent 1s onto vocab.pimpl
swa_layers[ 0 .. 511 ]  legitimate array (2048 B)
        ...
swa_layers[k]   = 1  ─┐
swa_layers[k+1] = 1  ─┴─►  vocab.pimpl = 0x1`00000001
        ...
   writes continue into  loras  (set buckets)

The address is forced, not chosen Because the primitive can only deposit 0s and 1s, the only non-trivial pointer two adjacent slots can form is 0x0000000100000001. We don't get to pick a convenient address — the bug picks it for us.

Sizing the write n_layer = 3852 is the OOB-write length — the number of 0/1 words we stamp past the array, enough to overflow all the way through both vocab.pimpl and the loras buckets the destructor walks. swa_period = 6 only places the rare 0s, so the two slots over pimpl are both 1 and the loras bytes stay usable.

Why we can ever reach 0x100000001 Only because the engine is non-PIE: the heap starts at a low, fixed base, so allocating our way upward — a multi-GB spray (§5) — eventually maps the ~4 GB mark and lets us place data exactly at 0x100000001. In a PIE build the heap lives at a different, randomised range: that address is never mapped, so dereferencing the corrupted pimpl just segfaults. CWE-822.
