Tick C — symlink(2) lands on disk

The libuv pool is processing F5 and S6 in parallel on different worker threads. Both syscalls are fast, but the symlink chain is the only one that can already mutate the filesystem.

Worker thread #2 runs symlink("/tmp/pwned", "dist/aaaaa") kernel creates a directory entry pointing to /tmp/pwned — no further checks
↓
Filesystem state mutated dist/aaaaa is now a symlink. Anyone opening that path with default flags will follow it.
↓
Completion signal → main thread S chain's .then() resolves — chain heads toward post-write utimes (skipped: x.type !== 'file')

Meanwhile F5's realpath may also have completed — but its callback merely plans to submit writeFile. The symlink already exists when that submission happens.
